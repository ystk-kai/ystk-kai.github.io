命名規則​

チャプター​

  • chapter_00 から開始する
  • 共通ルートは chapter_00 から chapter_04 まで連番
  • 個別ルートは chapter_11 以降の別帯域に分ける
  • chapter_10 番台はヒロイン個別ルート専用にする

シーン​

  • scene_01_* のように2桁連番を使う
  • ファイル名末尾は内容が分かる短い英語にする
  • 1ファイル1ラベルではなく、1ファイル = 1ドラマ単位 を原則にする

ドラマ単位の目安​

  • 導入から選択肢、その分岐後の短い返りまでを1つにまとめる
  • 同じ場所、同じ感情課題、同じ会話テーマなら1ファイルに含める
  • 場所や感情課題が切り替わるところで分割する

例​

  • scene_01_after_school.md
    • 放課後の校舎
    • 旧音楽室前のためらい
    • 最初の遭遇
  • scene_02_first_response.md
    • 最初の返答選択
    • 会話の返り
    • 帰路の独白
  • scene_03_next_morning.md
    • 翌朝の再会
    • 次章への接続

index.md の責務​

各章ディレクトリの index.md には、章メタ情報だけを持たせる。

役割​

  • id
  • title
  • 読み込み順
  • 章全体の注意書き

例​

---
id: chapter_01
title: 第一章 - 昼の顔、夜の顔
---

本文は原則入れない。 本文まで index.md に書き始めると、再び巨大ファイルへ戻るためである。

シーンファイルの責務​

  • 各 scene_yy_*.md は、1ドラマ単位の本文を持つ
  • frontmatter は持たない
  • ファイル内に複数の # label を含んでよい
  • 章をまたぐ責務は持たせない

例​

# after_school

@bg school_corridor_evening fade 2000
@bgm empty_school fadeIn=1500

> 放課後の校舎は、音が消えていくのが分かる。

# first_meeting

@bg old_music_room_interior dissolve 1200

コンパイラの責務​

コンパイラは、以下の流れで動くべきとする。

  1. scenarios/ 配下を走査する
  2. chapter_* ディレクトリを検出する
  3. 各ディレクトリの index.md を読み込む
  4. scene_*.md をファイル名順で連結する
  5. 1つの ScenarioChapter として出力する

ランタイム側は、個々のシーンファイルを意識しない。 シーン分割は執筆とレビューのためであり、実行単位は従来通り ScenarioChapter でよい。

docs との対応​

物語資料との対応は以下で揃える。

docsscenarios
story-treatment.md章とルートの大筋
scene-breakdown.mdscene_yy_*.md の分割基準
chapter-outline.mdchapter_xx 単位の確認
実シナリオscenarios/chapter_xx_*/scene_yy_*.md

運用ルール​

  • 新規執筆は必ず scene_yy_*.md で行う
  • 章全体の調整は index.md ではなく docs 側で行う
  • ルートの追加は route_* を増やさず、新しい chapter_xx_* を切る
  • レビュー時は scene_yy_*.md 単位で差分を見る
  • 章の順序変更は、番号と docs 側の両方を更新する

決定記録 -- 設計決定の背景と経緯

解決したい問題​

  • chapter01.md のような大きい単一ファイルは、差分が追いづらい
  • 共通ルートと個別ルートの責務がファイル名から分かりにくい
  • プロローグだけ命名規則が外れている
  • シナリオレビュー時に、どのシーンを直したか識別しづらい
  • 将来的に分岐や追加ルートが増えると、フラット配置では破綻する

採用方針​

  • 章はすべて chapter_xx_* 形式で統一する
  • プロローグも chapter_00 として扱う
  • ルートも route_* ではなく、個別の chapter_xx_* として扱う
  • 原本は章ディレクトリ配下の scene_yy_*.md に分割する
  • コンパイラは章ディレクトリを1つのシナリオチャプターとして扱う

命名規則の理由​

  • chapter_00 でプロローグを例外扱いしない
  • chapter_01 から chapter_04 までで共通ルートが分かる
  • chapter_11 chapter_12 chapter_13 を見れば、どのヒロインルートか一目で分かる
  • 将来 chapter_21_after_story のような追加シナリオ帯域も確保しやすい

分けすぎない理由​

  • ラベルごとに切ると文脈が切れすぎる
  • 前後の会話の呼吸を確認しづらい
  • レビュー単位としては、ラベルよりも会話のまとまりの方が自然である

移行方針​

以下は設計時点の移行計画である。第1〜2段階は完了済み。

第1段階 (完了)​

  • docs 側で本資料を基準化する
  • story-treatment.md と scene-breakdown.md を正本にする

第2段階 (完了)​

  • scenarios/ を章ディレクトリ構成へ分解する
  • 先に chapter_00 から chapter_04 の共通ルートを移す

第3段階​

  • ルートファイルを chapter_11 以降へ移す
  • コンパイラと GameScreen.tsx の読込方式を更新する

第4段階​

  • 旧フラット構成を退避または削除する

現時点では旧構成を scenarios_flat_legacy/ に退避している。

この構成の利点​

  • 1シーン単位でレビューできる
  • 共通ルートと個別ルートの責務が明確になる
  • プロローグだけ命名が浮かない
  • 将来のアフターやIFルートを番号帯域で増やせる
  • 物語資料と実装の対応が取りやすい
