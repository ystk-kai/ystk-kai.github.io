ミニゲーム実装設計

目的​

本資料は、ミニゲーム収集システムを既存のノベルエンジンとタイトルUIへ統合するための実装設計を定義する。 対象は次の三点である。

  • ストーリー中でミニゲームを発見・解放する導線
  • 解放状態とメダル進行のグローバル保存
  • タイトル画面からの再挑戦とミニゲーム一覧内の隠しEND解放

実装方針​

1. スロットセーブとは分離する​

  • ミニゲーム解放状態は GlobalClearData に保持する
  • セーブスロットに紐づけない
  • セーブを消しても、発見済みミニゲームとベストスコアは残る

2. ストーリー実行とミニゲーム実行を分ける​

  • シナリオ中は 発見 のみ行い、実際のプレイはタイトル画面の休憩所に寄せる
  • タイトルからのミニゲーム実行時は ScenarioRunner を使わない

3. まず基盤を共通化する​

  • 各ゲーム個別に状態管理を持たせすぎない
  • 開始 / 終了 / メダル付与 / 記録更新 の共通インターフェースを切る
  • UI殻を先に作り、その中に個別ゲームを差し込む

追加するデータ構造​

GlobalClearData を以下のように拡張する。

export interface MinigameRecord {
  unlockedAt?: number;
  playCount: number;
  bestScore: number;
  bestMedalPayout: number;
  lastScore: number;
  lastMedalPayout: number;
  lastPlayedAt?: number;
}

export interface GlobalClearData {
  version: string;
  clearedRoutes: string[];
  clearedEndings: string[];
  firstClearTimestamp?: number;
  playthroughCount: number;
  currentMedals: number;
  lifetimeMedals: number;
  unlockedMinigames: string[];
  minigameRecords: Record<string, MinigameRecord>;
  unlockedBonusScenes?: string[];
}

バージョン運用​

  • GLOBAL_CLEAR_VERSION は 1.2.0 へ更新する
  • 読み込み時に unlockedMinigames と minigameRecords が無ければ初期値補完する
  • 既存ユーザーのクリアデータを壊さない後方互換を維持する

ミニゲーム定義マスター​

アプリ側に静的なマスターを置く。

想定配置:

  • apps/web/src/data/minigameDefinitions.ts

想定内容:

export interface MinigameDefinition {
  id: string;
  title: string;
  description: string;
  unlockSceneId: string;
  rewardType: "skill" | "card" | "luck";
  hiddenEndEligible: boolean;
  runtime: "story_only" | "title_replay" | "both";
}

このマスターを単一の真実源とし、 タイトル一覧表示、解放判定、隠しEND条件判定に使い回す。

SaveManager 拡張​

packages/engine/src/SaveManager.ts に以下の責務を追加する。

  • unlockMinigame(minigameId: string): void
  • isMinigameUnlocked(minigameId: string): boolean
  • recordMinigameResult(minigameId: string, score: number, medalPayout: number, options?): void
  • addMedals(amount: number): void
  • spendMedals(amount: number): void
  • awardMedals(amount: number, lifetimeGain?: number): void
  • getUnlockedMinigames(): string[]
  • getMinigameRecord(minigameId: string): MinigameRecord
  • hasReachedLifetimeMedals(target: number): boolean
  • unlockBonusScene(bonusId: string): void
  • isBonusSceneUnlocked(bonusId: string): boolean

記録ルール​

  • 初回起動時に unlockMinigame を呼ぶ
  • 終了時に recordMinigameResult を呼ぶ
  • 無料台は終了時に currentMedals と lifetimeMedals の両方へ反映する
  • 賭け台は spendMedals で所持メダルを即時減算し、払い戻し時だけ awardMedals で戻す
  • lifetimeMedals は賭け消費では減らさず、純増分のみ加算する
  • bestScore と bestMedalPayout は別管理にする

ストーリーからの解放導線​

今回は新コマンドを増やさず、 特定の chapterId + sceneLabel に到達したら解放する 方式を採る。

理由​

  • ストーリー側の改修量を抑えられる
  • タイトル休憩所という役割に対して、発見だけできれば十分
  • ノベル進行とミニゲーム進行を強く結びつけずに済む

Zustand ストア拡張​

apps/web/src/store/gameStore.ts では、フェーズとミニゲーム操作を追加する。

追加フェーズ​

  • minigame_menu
  • minigame_playing
  • bonus_scene_playing

追加状態​

  • selectedMinigameId: string | null
  • minigameContext: "story" | "title" | null
  • minigameResult: { score: number; medalPayout: number; isNewRecord: boolean } | null

追加アクション​

  • openMinigameMenu(): void
  • startMinigame(minigameId: string, context: "story" | "title"): void
  • finishMinigame(result): Promise<void>
  • refreshGlobalClearData(): void
  • checkMinigameUnlock(chapterId: string | null, sceneLabel: string | null): void

タイトル画面変更​

apps/web/src/components/ui/TitleScreen.tsx に ミニゲーム を追加する。

表示ルール​

  • 1本も未解放なら disabled か非表示
  • 1本以上解放で選択可能
  • 隠しENDはタイトル直下へは出さない

遷移先​

  • ミニゲーム → minigame_menu

ミニゲーム選択画面​

新規コンポーネント案:

  • apps/web/src/components/ui/MinigameMenu.tsx

表示項目:

  • タイトル
  • 解放元の簡易ヒント
  • ベストスコア
  • ベスト獲得メダル
  • プレイ回数
  • 所持メダル / 累計メダル

重要なのは、未発見のものを全部ネタバレしないことだ。 そのため ??? と簡易シルエット程度で止める。

隠しEND解放ロジック​

条件:

  1. hiddenEndEligible = true のミニゲームがすべて解放済み
  2. lifetimeMedals が目標値以上

達成時の動作:

  • unlockBonusScene("minigame_hidden_end")
  • MinigameMenu の末尾に 放課後ゲームコレクター を表示
  • 選択時、本編と同じ ScenarioRunner で専用章を再生する
  • ボーナスENDの開始は checker_out、終了は章末の checker_in で通常ADVの終端演出に合わせる
  • ボーナスENDの本文は scenarios/ 配下の独立ディレクトリとして管理し、scenario-compiler 経由で apps/web/src/scenarios/ に出力する

エンディング管理​

  • ending_minigame_hidden を ENDING_FLAG_MAP に追加する
  • routeId は bonus_minigame のような独立IDにする
  • メイン3ルートやTrue解放条件には含めない

ミニゲーム個別実装の共通IF​

各ミニゲームコンポーネントは以下の props を満たす形に揃える。

export interface MinigameProps {
  onFinish: (result: { score: number; medalPayout: number }) => void;
  currentMedals: number;
  initialSeed?: number;
}

これにより、ゲーム本体とタイトルUIの結合を弱く保てる。

推奨ディレクトリ構成​

apps/web/src/minigames/
  components/
    MinigameShell.tsx
  definitions/
    arcadeStack.ts
    card21.ts
    luckyReels.ts
  games/

テスト方針​

エンジン​

  • GlobalClearData 拡張後も既存データを読める
  • unlockMinigame が重複登録しない
  • recordMinigameResult がスコアとメダルを正しく更新する
  • 累計メダル判定が定義値と一致する

ストア​

  • シーン到達時に対応ゲームだけ解放される
  • 終了後に globalData とメダル表示が更新される
  • タイトル経由のプレイで scenario flags が汚れない

UI​

  • 未解放状態で ミニゲーム が押せない
  • 解放後に一覧へ遷移できる
  • 条件達成で一覧末尾に隠しENDが表示される
  • 所持メダルと累計メダルが正しく表示される

実装順序​

  1. GlobalClearData の拡張
  2. SaveManager の新API追加
  3. gameStore のフェーズ追加
  4. タイトル画面とミニゲーム一覧の追加
  5. シーン到達での解放判定追加
  6. arcade_stack を最初の実ゲームとして実装
  7. card_21 と lucky_reels を共通殻の上に追加
  8. ミニゲーム一覧内の隠しEND解放処理を実装

実装上の注意​

  • タイトル再挑戦時は、バックログや本編BGM状態を持ち込まない
  • ミニゲーム失敗時も無限ループ導線にしない
  • タッチ操作を考え、キーボード専用に寄せすぎない
  • card_21 と lucky_reels は賭け額の選択が直感的で、残高不足時の詰まりが見えやすい UI にする
  • ボーナスENDはギャグ寄りにしつつ、湊・いろは・あかりの既存イメージを崩さない
